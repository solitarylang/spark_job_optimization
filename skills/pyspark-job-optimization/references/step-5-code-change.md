# 第 5 步：代码变更

## 目的

只执行用户已经批准的优化。

## 输入

- Approved optimization items
- Current source files

## 流程

1. 先保留优化前代码快照，不依赖 git 作为版本记录。
2. 只改最小必要的代码面。
3. 除非优化明确要求改变行为，否则保持行为一致。
4. 保持可读、可审查。
5. 把修改后的路径再和原瓶颈核一遍。
6. 如果修复会碰到多个热点，只有在它们共享同一个根因时才合并处理。
7. 完成后保留优化后代码快照，确保 `source/` / `source_before/` 和 `source_optimized/` / `source_after/` 都能回溯。

## 输出

- 更新后的代码
- 简短变更摘要
- 仍未解决的问题（如有）
- 修改路径的验证说明
- 优化效果摘要
- 本次优化沉淀的经验点
- 优化前 / 优化后代码快照位置说明

## 代码快照要求

- 如果还没有优化前快照，先把当前工作副本复制到 `source_before/` 或 `source/` 的稳定备份目录。
- 修改完成后，再把最新代码复制到 `source_optimized/` 或 `source_after/`。
- 这两个目录必须同时保留，作为后续对比的唯一代码版本来源。
- 不要用 git 历史代替代码快照目录，也不要只保留单个版本。
