# 第 3 步：根因排序

## 目的

结合源码阅读和运行证据，给主要慢因排序。

## 输入

- Code hotspots from step 1
- Runtime evidence from step 2

## 流程

1. 把症状对应到代码路径。
2. 把直接原因和次级症状分开。
3. 如果多个问题最终能靠同一个修复解决，就合并成一个原因。
4. 按影响度排序。
5. 每项只标 `已确认` 或 `待确认`。

## 输出

- 前 5 个原因
- 每个原因对应的 stage
- 每个原因对应的代码范围
- 支持每个原因的执行证据
- 还缺哪些证据
- 为什么这个排序优先于其他方案
- 是否存在 skew / 长尾 / 极端大分区

## 强制规则

- 第 3 章必须建立在第 1 章的 `源码 -> stage -> 指标` 对照基础上。
- 不能只按文件或函数名排序，必须先说清楚对应哪个 stage。
- 每个根因都要能回指到具体 stage 的耗时、任务数、shuffle、失败或 spill 证据。
- 如果 stage 总量不大但仍然很慢，要优先检查 skew、长尾和热点 key，而不是先把它归到总量问题。
- 观察到但无法确认的内容只能放在观察项，不进入主根因列表。
