# 第 3 步：根因排序

## 目的

结合源码阅读和运行证据，给主要慢因排序。

## 输入

- Code hotspots from step 1
- Runtime evidence from step 2

## 流程

1. 把症状对应到代码路径。
2. 把直接原因和次级症状分开。
3. 如果多个问题最终能靠同一个修复解决，就合并成一个原因。
4. 按影响度排序。
5. 每项只标 `已确认` 或 `待确认`。
6. 源码侧如果发现上游有字段、下游没用、最终也没写入结果表，要优先当作冗余字段观察项，不要直接算成主慢因。

## 输出

- 前 5 个原因
- 每个原因对应的 stage
- 每个原因对应的代码范围
- 支持每个原因的执行证据
- 还缺哪些证据
- 为什么这个排序优先于其他方案
- 是否存在 skew / 长尾 / 极端大分区

## 强制规则

- 第 3 章必须建立在第 1 章的 `源码 -> stage -> 指标` 对照基础上。
- 不能只按文件或函数名排序，必须先说清楚对应哪个 stage。
- 每个根因都要能回指到具体 stage 的耗时、任务数、shuffle、失败或 spill 证据。
- 冗余字段如果没有直接运行证据，只能作为观察项或优化候选，不能直接排进主慢因。
- 如果某个 stage 的读取行数超过 10 亿，或者输入规模超过 2T，先单独判断这个量级是否合理；不合理时优先按数据量和粒度问题处理，而不是直接归到普通慢因。
- 如果读取的数据量不大，但文件数很多，或者平均单文件大小小于 16M，先单独判断是否存在小文件问题；这类问题优先按上游小文件、合并缺失或写入粒度过细处理，而不是直接归到普通慢因。
- 如果发现的是冗余中间列，尤其是排序字段、`rank` / `row_number` / `dense_rank` 这类下游未消费的列，优先把它当成“应删除该列计算逻辑”的优化项，而不是只保留末端 drop。
- 如果 scan 的是 130 天、360 天这类长历史分区全量表，优先把它当成中间表、增量计算或解析结果复用问题来评估，而不是只看单次慢不慢。
- 如果同一条业务链在不同日期执行时都会重新解析同一批历史数据，要优先作为重复执行问题评估，通常比单次 scan 慢因更值得先改。
- 如果超大 stage 超过 4 个，且每个 stage 都有明显耗时或重 shuffle，要优先考虑部分 stage 落表 / 中间表化，而不是只做单点参数调优。
- 如果 stage 总量不大但仍然很慢，要优先检查 skew、长尾和热点 key，而不是先把它归到总量问题。
- 如果窗口排序、groupBy、join 这类 shuffle 慢点出现时，先判断是总量问题还是倾斜问题，再决定是否落表、裁剪或调参。
- 观察到但无法确认的内容只能放在观察项，不进入主根因列表。
