# 最终报告规范

## 目标

最终报告固定按 5 章输出，对应完整优化流程：

1. 源码分析
2. 运行日志分析（优化前）
3. 根因分析
4. 优化方案
5. 优化效果验证（优化后）

每章只写自己的证据和结论，不提前越界。

## 总体要求

- 第 1 章主要由源码分析子 skill 产出。
- 第 2 章主要由 Spark UI / 日志分析子 skill 产出，建议用表格或流程图组织运行链路。
- 第 3 章基于前两章做根因排序。
- 第 4 章基于根因输出优化方案和待确认项。
- 第 5 章补优化后验证、前后对比和经验沉淀。
- `output/<case_name>/` 下保留第 4 章单独的文字版和图片版。
- 版式优先参考 `references/final-report-template.md`。
- 第 4 章细节和图片继续参考 `references/step-4-report-template.md` 与 `references/assets/step4-report-sample.svg`。
- 第 5 章对比基于 `spark_ui/browser/` 和 `spark_ui/optimized_browser/`。

## 每章必须产出

### 第 1 章：源码分析

- `源码 -> stage -> 指标` 对照表
- 按 stage 划分的代码路径
- 每个 stage 的算子链路、风险点和观察项
- `已确认` / `待确认`

### 第 2 章：运行日志分析（优化前）

- 集群与环境信息
- Jobs / Stages / Executors / SQL / EventLog / AM / Driver 证据；如果能直接定位到 SQL，要同时保留 SQL 原文、执行计划和对应 stage
- 关键 stage 的耗时、任务数、shuffle、失败、spill
- 运行链路表或流程图，用于串起整体执行过程
- `已确认` / `待确认`

### 第 3 章：根因分析

- 前五个根因
- 每个根因对应的 stage 和代码范围
- 每个根因的证据、影响度和待确认项

### 第 4 章：优化方案

- 参数调优候选
- 实现方式优化
- 业务逻辑优化
- 每项优化的方向、待确认项、预期收益估算逻辑，以及对应的代码调整片段或代码草图
- 只有在有直接证据时，才写具体风险；如果要写风险，必须说清楚会影响什么。没有证据的收口 / 写出动作不要单列成风险。

### 第 5 章：优化效果验证（优化后）

- 优化后应用信息
- 优化前后关键指标对比
- 运行时长、job 数、stage 耗时、shuffle、失败数、dead executor 对比
- 优化前 / 优化后代码快照对比
- 效果结论和剩余待确认项
- 本次优化沉淀出来的新经验和可复用规则

## 状态和收益口径

- 只写 Spark UI / Spark 日志直接暴露的信息。
- 没有暴露的字段写 `待确认` 或 `未直接给出`。
- 收益只写 `高 / 中 / 低`、节省秒 / 分钟 / 小时、减少 shuffle 量、减少扫描次数；不能估算就写 `待确认`。

## 图片要求

- 第 4 章仍然只输出一张总览图，而且每个优化热点都必须在图里带上预期收益估算和代码调整片段或代码草图。
- 图里若出现风险描述，必须是具体风险，不要把没有证据的收口动作写成风险。
- 对于窗口排序、`groupBy`、`join` 这类 shuffle 慢点，图里的 `待确认原因` 必须先判断是总数据量过大还是数据倾斜更严重，不能只写“很慢”。
- 左侧必须展示完整源码，并按 Spark stage 用虚线框拆分代码块。
- 每个 stage 代码块必须展示执行状态或 `待确认`，且文字必须完整落在框内，不能压线、不能越框。
- 右侧必须放热点放大镜卡片，卡片文字同样不能压线、不能越框。
- 同一热点的卡片之间必须保留足够间距，后一张卡的标题不能覆盖前一张卡的内容。
- 图里的说明文字必须是中文，代码和命令保留原样。
- 除代码和命令外，不允许出现猜测性文字。
