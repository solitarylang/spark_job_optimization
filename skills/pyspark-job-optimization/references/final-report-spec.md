# 最终报告规范

## 目标

最终优化报告需要回答三个问题：

1. 执行日志直接支持了哪些代码位置？
2. 日志里能直接读到哪些当前运行状态？
3. 因为证据不足，哪些内容仍然是 `待确认`？

## 必需输出

- 报告顶部放一段简短总结，只能用源码和执行日志里直接存在的证据说明这个任务在做什么
- `output/<case_name>/` 下放一张总览图
- 只有在需要解释图片时，才补一段短文本
- 放一个 `总结` 小节，只写直接证据，并把未暴露项标成 `待确认`
- 具体版式必须参考 `references/step-4-report-template.md`
- 图片生成必须参考 `references/assets/step4-report-sample.svg` 和 `references/assets/step4-report-sample.png`

## 优化优先级

所有建议都必须按这个顺序排列：

1. 先调 Spark 执行参数
2. 再改实现方式，但不改业务逻辑
3. 只有前两类不够时，才考虑改业务逻辑

在前两类方案没有用完之前，报告不能先写业务逻辑改动。

## 瓶颈卡片字段

每个瓶颈项必须包含：

- `排名`
- `代码范围`
- `日志证据`
- `待确认原因`
- `待确认收益`
- `优化方向`
- `待确认改动`
- `预期修改代码`

只有在执行日志和源码都能清楚验证时，才给出预期修改代码块。如果改动不确定，就写 `待确认改动`，但仍然要给出大致的 `优化方向`，不要把优化项留空。

## 状态定义

只总结导出的 Spark 日志或 Spark UI 明确暴露出来的当前运行状态：

- stage / job 编号
- 耗时
- 任务数
- 读取行数 / 记录数（如有）
- shuffle 读 / 写
- 重试 / 失败 / spill / preemption

如果没有暴露行数 / 记录数，就明确写成不可用，不要推断。

## 收益定义

收益只允许写成下面几种形式：

- `高 / 中 / 低`
- 预计节省秒 / 分钟 / 小时
- 减少 shuffle 量
- 减少扫描次数

如果没有精确数值，就把该项标成 `待确认`，不要估算。

## 图片规则

- 保持为单张图。
- 不看上下文正文也能读懂。
- 代码范围和运行证据要放在一起。
- 如果源码长度适合完整展示，优先用完整代码背景加放大镜热点标注。
