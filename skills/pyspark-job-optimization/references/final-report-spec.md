# 最终报告规范

## 目标

最终优化报告按 4 个章节组织，对应流程中的第 1 到第 4 步：

1. 源码分析
2. 运行日志分析
3. 根因分析
4. 优化方案

每个章节都只承接自己那一步的输出，不要把前后步骤混在一起。

## 必需输出

- 最终报告正文按 4 章输出，每章职责清晰、边界分明。
- `output/<case_name>/` 下保留第 4 章单独的文字版和图片版。
- 如果需要总览图，可以额外提供，但不能替代四章正文。
- 具体版式必须参考 `references/final-report-template.md`。
- 第 4 章的细节和图片必须继续参考 `references/step-4-report-template.md` 与 `references/assets/step4-report-sample.svg`。

## 四章分工

### 第一章：源码分析

- 只写源码里直接可见的执行路径、逻辑关系、数据流和潜在风险。
- 必须指出可能造成 `scan` 放大、`shuffle` 放大、增行、冗余列传递或重复聚合的位置。
- 不要把 Spark UI 结论提前写进这一章。

### 第二章：运行日志分析

- 只写 Spark UI、Spark 日志、YARN / AM / driver / executor 日志里直接可见的证据。
- 必须单独整理环境、jobs、stages、executors、SQL、AM / driver / executor 失败信息。
- 不能没有证据就补因果。

### 第三章：根因分析

- 只能基于前两章的事实和已有经验规则进行归因。
- 影响最大的前五个问题按证据强度排序。
- 观察到但证据不足的内容只写观察项，不进入主瓶颈。

### 第四章：优化方案

- 必须按 `Spark 参数 -> 实现方式 -> 业务逻辑` 的顺序输出。
- 没有明确可落地代码时，也必须给出大致优化方向。
- 只有源码和日志都能清楚验证时，才给预期修改代码。
- 这一章的正文、卡片字段和图片布局都要继续遵守 `step-4-report-template.md`。

## 每章必须产出

### 第一章：源码分析

- `源码 -> stage -> 指标` 对照表
- 按 stage 拆分的源码路径
- 每个 stage 的算子链路
- 每个 stage 的风险点和观察项
- 这一章主要由源码分析子 skill 产出

### 第二章：运行日志分析

- 集群与环境信息
- Jobs / Stages / Executors / SQL / EventLog / AM / Driver 证据
- 关键 stage 的耗时、任务数、shuffle、失败、spill
- 这一章主要由 Spark UI / 日志分析子 skill 产出

### 第三章：根因分析

- 前五个根因
- 每个根因对应的 stage 和代码范围
- 每个根因的证据、影响度和待确认项

### 第四章：优化方案

- 参数调优候选
- 实现方式优化
- 业务逻辑优化
- 每项优化的方向、风险和待确认项

## 状态定义

只总结导出的 Spark 日志或 Spark UI 明确暴露出来的当前运行状态：

- stage / job 编号
- 耗时
- 任务数
- 读取行数 / 记录数（如有）
- shuffle 读 / 写
- 重试 / 失败 / spill / preemption

如果没有暴露行数 / 记录数，就明确写成不可用，不要推断。

## 收益定义

收益只允许写成下面几种形式：

- `高 / 中 / 低`
- 预计节省秒 / 分钟 / 小时
- 减少 shuffle 量
- 减少扫描次数

如果没有精确数值，就把该项标成 `待确认`，不要估算。

## 图片规则

- 第 4 章仍然只输出一张总览图。
- 优先 `SVG`，必要时再补 `PNG`。
- 左侧必须放完整执行代码或完整代码链。
- 右侧必须放热点放大镜卡片。
- 代码和卡片之间要能直接回指。
- 图里的所有说明文字必须是中文，代码和命令保留原样。
- 除代码和命令外，不允许出现猜测性文字。

## 章节边界

- 第一章回答“源码在做什么、怎么做、有什么风险”。
- 第二章回答“运行时实际发生了什么”。
- 第三章回答“最主要的问题是什么、为什么这样排”。
- 第四章回答“应该怎么优化、先后顺序是什么、哪些还待确认”。
