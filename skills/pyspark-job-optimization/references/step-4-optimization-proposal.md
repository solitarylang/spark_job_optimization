# 第 4 步：优化建议

## 目的

把排序后的证据转成具体优化选项，并等待确认；如果日志不能直接支持某个结论，就写 `待确认`。

## 输入

- 前 5 个原因
- 当前代码约束
- 用户偏好或允许的改动范围

## 建议格式

- 原因
- 直接日志证据
- 拟定优化方案
- 优化方向
- `待确认` 项
- 优先级（`P0`、`P1`、`P2`）
- 风险或权衡（只写有直接证据支撑的具体风险，并说明会影响什么）
- 是否需要用户批准
- `预期收益`，必须写明估算逻辑，优先引用 stage 时长、shuffle、输入规模、任务数等数据
- `预期修改代码` 或 `预期修改代码（草图）`

## 优化顺序

提优化时，必须按这个顺序排序：

1. 先调 Spark 执行参数
2. 再改实现方式，但不改业务逻辑
3. 只有前两类不够时，才考虑改业务逻辑

不要一上来就写业务逻辑改动。

## 输出

- 按顺序排列的优化列表
- 等待确认的条目
- 可直接应用的条目
- 按预期影响度排序，其次按风险排序
- 包含一个简短 `总结` 小节，说明这个任务在做什么、核心逻辑是什么
- 这一节将作为最终报告的第 4 章，因此正文必须贴近 `references/step-4-report-template.md`
- 第 4 章图片的版式必须参考 `references/assets/step4-report-sample.svg` 和 `references/assets/step4-report-sample.png`
- 当 scan 量过大时，可以把“构建中间表 / 中间快照”作为一类标准化建议
- 当 scan 的是 130 天、360 天这类长历史分区全量表，或者同一批历史数据会反复执行时，可以把“中间表 / 增量计算 / 解析结果复用”作为一类标准化建议
- 当超大 stage 数量超过 4 个，且这些 stage 共同构成一条长链路时，可以把“部分 stage 落表 / 中间表化”作为一类标准化建议，避免任务中途失败后前面已成功的 stage 数据无法复用
- 当 join 链路过长或重复 join 很多时，可以把“按天构建中间表”作为一类标准化建议
- 当输入 scan 进来的字段或分区最后没有用到时，要把“末端裁字段 / 裁分区”作为一类标准化建议，避免前面读得多、最后用得少
- 当发现冗余中间列，尤其是排序字段、`rank` / `row_number` / `dense_rank` 这类下游未消费的列时，要把“删除该列的计算逻辑”作为标准化建议，而不是只做末端 drop
- 如果没有明确可落地的最终代码，也必须给出可执行的代码草图或局部 diff，不能只写一句优化方向
- `预期收益` 不要空着；如果不能精确到分钟，也要说明是基于哪些 stage 指标推出来的，并给出高 / 中 / 低 级别判断
- 没有直接证据支撑的收口动作、末端写出动作或常规输出动作，不要单独写成风险；如果确实有风险，必须写清楚具体影响是什么。
- 对于窗口排序、`groupBy`、`join` 这类 shuffle 慢点，优化方案里的 `待确认原因` 必须先区分总量和倾斜，再决定是落表、裁剪还是调参。

## 报告图片

第 4 步报告必须输出为 `output/<case_name>/` 下的一张图，除代码和命令外，所有描述句都必须能被执行日志直接支持。

### 推荐格式

- 优先使用一张 `svg` 图。
- 只有在需要位图分享时才补一张 `png`。
- 只保留一张总览图，不要拆成多张。

### 图片内容

图片必须同时展示代码项和直接日志证据；如果某个字段不是日志直接暴露的，就标成 `待确认`。

推荐版式是：

- 左侧放完整执行代码或完整代码链
- 用虚线框把代码按 Spark stage 拆开
- 每个 stage 框都要显示执行状态或 `待确认`，并且文字必须完整落在框内，不能压线、不能越框
- 在代码里标出 `TOP5` 瓶颈热点
- 每个热点配一个放大镜式说明卡
- 直接日志证据、剩余 `待确认` 项、以及必须展示的预期修改代码或代码草图

每张瓶颈卡片必须包含：

- `排名`
- `代码范围` 和行号范围
- `stage 范围` 和对应执行状态
- `日志证据`
  - stage / job 编号
  - 耗时
  - 任务数
  - 读取行数 / 记录数（如有）
  - shuffle 读 / 写
  - retry / failed / spilled / preempted 信号（如有）
  - 如果没有暴露行数 / 记录数，就标明不可用
- `待确认原因`
- `预期收益`
- `待确认改动`

每个进入优化列表的热点都必须给出代码调整片段；如果最终代码不能完全验证，就给出可执行的代码草图或局部 diff，并在文字里标清 `待确认`。

### 版式要求

- 保持图片作为单张总览图仍然可读。
- 高影响项要比低影响项更突出。
- 保留代码上下文，不要只留摘要标签。
- `TOP5` 项用卡片说明，风格可以参考旧 skill 的 stage mapping 图，但对象要换成瓶颈代码。
- stage 的虚线框必须把对应代码块包起来，不能只写文字标签。
- 优先把原始代码上下文和修改草图放在一起，而不是只放摘要标签。
- 如果文字可能越界，优先增加留白或缩小字号，不要让文字压到边框上。
- 改动没有完全验证时，不要强行塞代码。
- 如果一个瓶颈跨多个 stage，就在主卡里放最慢的 stage，其他 stage 作为辅助证据。
- 代码主干或代码链上下文必须和卡片一起可见。
- 不要把证据藏在抽象标签后面；每张卡都要能回指到真实代码范围和真实运行状态。
- 每张卡都要给出预期收益估算，不要留空；如果不能精确量化，也必须写清估算逻辑和引用的 stage 指标。

### 旧 skill 风格借鉴规则

- 只保留一张总览图。
- 高影响区域要视觉上更突出。
- 辅助信息要紧凑、可读。
- 版式要稳定，方便不同 case 之间对比。
