# 最终优化报告模板

这是整份最终优化报告的总模板。五个章节分别对应流程中的第 1 到第 5 步，后续所有案例都应尽量贴近这个结构。

## 目录结构

```text
output/<case_name>/
  final_optimization_report.md
  final_optimization_report.svg
  final_optimization_report.png
  step4_top5_bottlenecks.md
  step4_top5_bottlenecks.svg
  step4_top5_bottlenecks.png
  context/
    context_report.md
  spark_ui/
    browser/
    ...
```

## 页面结构

```text
# 最终优化报告

案例：`<case_name>`

## 第一章：源码分析
### 1.1 输入与执行路径
### 1.2 主要逻辑与数据流
### 1.3 按 stage 拆分的源码对照
### 1.4 潜在风险与增行点

## 第二章：运行日志分析（优化前）
### 2.1 集群与环境信息
### 2.2 Jobs / Stages / Executors
### 2.3 SQL / EventLog / AM / Driver 证据

### 2.4 长尾 / skew / 失败链路

## 第三章：根因分析
### 3.1 证据对齐
```

## 每章必须产出

| 章节 | 必须产出 | 说明 |
|---|---|---|
| 第一章：源码分析 | `源码 -> stage -> 指标` 对照表；按 stage 划分的代码路径说明；每个 stage 的算子链路；每个 stage 的风险点和观察项 | 主要由源码分析子 skill 产出，要求能直接看出每个算子对应的 stage 运行状况 |
| 第二章：运行日志分析（优化前） | 集群 / 环境信息；Jobs / Stages / Executors / SQL / EventLog / AM / Driver 证据；如果能直接定位到 SQL，要保留 SQL 原文、执行计划和对应 stage；每个关键 stage 的任务数、耗时、shuffle、失败、spill 信息；运行链路表或流程图 | 主要由 Spark UI / 日志分析子 skill 产出，需要保留长尾、skew、失败链路这些用于后续对比的指标 |
| 第三章：根因分析 | 前五个根因；每个根因对应的 stage 和代码范围；每个根因的证据、影响度和待确认项 | 基于前两章事实和经验规则做归因 |
| 第四章：优化方案 | 参数调优候选；实现方式优化；业务逻辑优化；每项优化的方向、风险、待确认项，以及预期收益的估算逻辑 | 必须按 `Spark 参数 -> 实现方式 -> 业务逻辑` 的顺序输出 |
| 第五章：优化效果验证（优化后） | 优化后应用信息；优化前后关键指标对比；运行时长、job 数、stage 耗时、shuffle、失败数、dead executor 对比；优化效果结论和剩余待确认项；本次优化沉淀出来的经验规则；以后可复用的诊断结论或检查清单 | 必须基于优化后重新采集的证据，并补充经验沉淀 |

## 填写要求

| 章节 | 填写要求 | 备注 |
|---|---|---|
| 第一章：源码分析 | 只写源码里直接可见的执行路径、逻辑关系和潜在风险；必须说明哪些位置会造成 scan 放大、shuffle 放大、增行、冗余列传递或重复聚合；不要把 Spark UI 结论提前写进这一章 | 结构应尽量按 stage 展开 |
| 第二章：运行日志分析 | 只写 Spark UI、Spark 日志、YARN / AM / driver / executor 日志里直接可见的信息；必须把环境、jobs、stages、executors、SQL 和失败证据分开写清楚；如果能定位到 SQL，要同步保留 SQL 原文、执行计划和对应 stage；建议用表格或流程图组织运行链路；如果某个值没有直接暴露，就写 `待确认` 或 `未直接给出`，不要补推断 | 主体应是优化前证据 |
| 第三章：根因分析 | 只能基于前两章的事实加上已有经验规则做归因；前五个问题按影响度排序，但不能把没有证据支撑的内容放进主排名；如果只是观察到但还不能确认，就进入观察项，不进入主瓶颈 | 不要把观察项混成主瓶颈 |
| 第四章：优化方案 | 必须按 `Spark 参数 -> 实现方式 -> 业务逻辑` 的顺序输出；每个优化项都必须给出代码调整片段或代码草图，并写明预期收益的估算逻辑，不能只写优化方向；只有有直接证据时才写风险，且要说明具体影响；这一章的详细结构应继续参考 `references/step-4-report-template.md` | 保留待确认项和风险 |
| 第五章：优化效果验证 | 只能基于优化后重新采集的 Spark UI / 日志 / eventlog 进行分析；必须和优化前运行证据做对比；如果优化后数据还没采集到，就写 `待确认`；这一章还要沉淀本次优化形成的新经验 | 要包含复盘和经验沉淀 |

## 图片要求

- 最终报告可以配一张总览图，但第 4 章的单独图片仍然保留。
- 第 4 章图片左侧必须展示完整源码，并按 Spark stage 用虚线框拆分代码块。
- 每个 stage 代码块都必须带执行状态或 `待确认`，且文字必须完整落在框内，不能压线、不能越框。
- 右侧每个热点必须展示 `预期收益` 和 `预期修改代码` 或 `预期修改代码（草图）`，不能只给优化方向。
- 同一热点的卡片之间必须保留足够间距，后一张卡的标题不能覆盖前一张卡的内容。
- 如果写风险，必须是具体风险，并说明会影响什么；没有证据的收口 / 写出动作不要单列成风险。
- 对于窗口排序、`groupBy`、`join` 这类 shuffle 慢点，卡片里的 `待确认原因` 必须先判断是总数据量过大还是数据倾斜更严重，不能只写“很慢”。
- 图片里的说明文字必须是中文，代码和命令保留原样。
- 如果文字可能越界，优先增加留白或缩小字号，不要让文字压到边框上。
- 不允许把推断性文字写成已确认结论。
